# Область видимости провайдеров

Объект, который виджет берёт из дерева (`context.read`/`watch`, `Provider.of`), должен раздаваться **предком** этого виджета. Скилл — про то, где этого предка ставить, как не потерять его при переиспользовании и переходах, и когда вместо чтения из дерева брать зависимость явно. Раздача контракта из DI-контейнера — в `conventions.md` и зональных скиллах; здесь — про то, что живёт в самом дереве через `Provider`.

## Главное правило

Провайдер `T` обязан быть выше по дереву **на каждом пути, где `T` читают** — не «где-то в приложении», а именно над этим местом. Нет предка — `ProviderNotFoundException` в рантайме, часто прямо в `build`.

## Где живёт зависимость

Вопрос «куда поставить провайдер» — это вопрос «сколько живёт объект». Две типичные области:

- **Уровень приложения (над роутером)** — живёт всю сессию приложения, доступен везде: настройки, навигатор, аналитика, тема. Ставится один раз при сборке, над `MaterialApp`.
- **Уровень экрана / сессии (под роутером)** — существует только пока открыт экран или идёт сессия: контроллер открытого документа, его модель, `BLoC` экрана. Создаётся при входе, `dispose` при выходе; **вне своей зоны его нет**.

Отсюда практический вывод: scoped-объект нельзя поднять на уровень приложения — до входа в экран ему нечего обслуживать (нет открытого документа, нет сессии). Если на каком-то экране такого объекта «не хватает» — чаще всего его там и не должно быть, а виджет просто переиспользован не на своём месте.

## Два способа потерять провайдера

1. **Переиспользование на чужом экране.** Виджет, завязанный на scoped-провайдер одной фичи, вставили туда, где этой фичи нет (например, экранный виджет — в глобальные настройки). Предка нет → падает. Чем «глобальнее» переиспользуемый виджет, тем опаснее держать в нём `context.read` фичевого объекта.

2. **Выход из поддерева через overlay/route/Hero.** `showDialog`, `showModalBottomSheet`, `push` нового маршрута и полёт `Hero` строят содержимое в **корневом** `Overlay`/`Navigator`, вне твоего поддерева провайдеров. Чтения внутри падают, даже если визуально это тот же экран.

## Как не падать

- **Переиспользуемому / межфичевому виджету — явная зависимость в конструкторе, а не `context.read`.** Неявное чтение проверяется только в рантайме; параметр конструктора заставляет каждый call-site передать зависимость (или осознанно занулить) — промах ловится при компиляции. Обязательная зависимость — `required`-параметр (см. `conventions.md`); зависимость, которой в части контекстов нет, — nullable-параметр со значением по умолчанию `null`, а живые эффекты под `?.`:

```dart
class StyleControls extends StatelessWidget {
  const StyleControls({super.key, this.engine}); // на экране без сессии — null

  final ReaderEngine? engine;

  void _apply(BookStyle style) {
    prefs.save(style);          // всегда: пишем настройку
    engine?.applyStyle(style);  // только когда сессия есть: живой эффект
  }
}
```

- **Когда виджет обязан читать из дерева** (глубоко внутри фичи, где предок гарантирован) — это нормально; в комментарии назови требуемого предка.

- **Через overlay/route — захвати зависимость у вызывающего и раздай заново через `.value`.** Контекст до `showDialog` ещё внутри твоего поддерева:

```dart
final engine = context.read<ReaderEngine>(); // берём в скоупе экрана
showDialog(
  context: context,
  builder: (_) => RepositoryProvider.value( // раздаём заново в корневом overlay
    value: engine,
    child: const StyleDialog(),
  ),
);
```

- **Для `Hero` — положи провайдеры ВНУТРЬ `Hero`.** Во время полёта шаттл — копия `child` героя, построенная в `Overlay`; провайдеры-`.value` внутри героя летят вместе с контентом, и чтения остаются валидными.

- **nullable-зависимость = «виджет работает и без сессии».** Guard живые эффекты через `?.`, скрывай секции, которым нужна сессия (`if (book != null) ...`); то, что просто пишет настройку, работает всегда.

## Диагностика «ProviderNotFoundException above X»

Запускать приложение не обязательно — диагностируется статически:

1. Найди все места, где конструируется `X` (grep по имени виджета).
2. На каждом проверь, есть ли провайдер искомого типа выше по дереву — или re-provide, если рядом `showDialog`/`showModalBottomSheet`/`push`/`Hero`.
3. Call-site без живого предка и есть баг.

## Связь со скиллами

- `conventions.md` — раздел UI (`context.read` для отправки события, `BlocBuilder`/`BlocSelector` для перестройки) и Конструкторы (обязательная зависимость — `required`-поле в init-list).
- `flutter-feature` — общее поднимаем выше; влияние на другую фичу — событием в её `BLoC` выше по дереву. Этот скилл — про механику раздачи такого «общего выше» через дерево.
- `flutter-navigation` — `AppNavigator` раздаётся `Provider`-ом из дерева (`context.read<AppNavigator>()`), а не из DI-контейнера: пример app-global зависимости в дереве.
