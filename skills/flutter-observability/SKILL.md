# Наблюдаемость

Как видеть, что происходит внутри приложения: события пользователя, ход работы, ошибки и падения. Зона живёт в `common/logging/`: наружу торчат контракты, конкретные реализации и SDK связываются при сборке. Объекты берёшь из DI-контейнера.

## Раскладка

`common/logging/` — зонтик, а не место для одного универсального сервиса:

- `observability/` — техническая видимость: логи, пойманные ошибки, warning/error/fatal, crash reporting, Sentry/Crashlytics/аналог.
- `analytics/` — продуктовые события: действия пользователя, экраны, воронки, Firebase Analytics/AppsFlyer/аналог.
- `bloc_observer.dart` — мост BLoC на верхнем уровне `logging/`; получает зависимости через конструктор.

Аналитику, ошибки и хранилище держи разными ответственностями.

## Три кирпича — когда какой

- **Логирование (`Logger`)** → диагностика для разработчика: ход выполнения, входы во внешние вызовы, пойманные ошибки и неожиданные состояния. Сообщения «для нас», в проде приглушаются.
- **Аналитика (`Analytics`)** → продуктовые события для бизнеса: что сделал пользователь — открыл экран, нажал, прошёл шаг. Стабильные имена событий, иногда с параметрами.
- **Наблюдатель BLoC** → авто-мост: сам следит за переходами состояний и шлёт их в логирование/аналитику, чтобы не расставлять вызовы вручную по коду. Связывается один раз при сборке.

Как завести контракты (`ErrorReporter`, `Analytics`) и каркас наблюдателя — `references/contracts.md`.

## Что логировать vs что трекать

- **Логируешь** — для отладки: технические подробности, ветви ошибок, стектрейсы.
- **Трекаешь** — для продукта: ключевые действия и вехи пользователя; без технических деталей и без персональных данных.
- Запах путаницы: шлёшь стектрейс как событие аналитики или логируешь «нажал кнопку» как продуктовое событие — поменяй местами.

## Ошибки и падения

- Пойманные ошибки — в логирование; на границе данных их маппят во внутренний тип ошибки приложения (`AppFailure` или проектный аналог).
- Падения и непойманные исключения — во внешний сервис отчётов о сбоях, спрятанный за контрактом. Какой именно сервис — деталь реализации, выбор под контекст проекта.

## Границы

- Контракт наружу, реализация и SDK внутрь: UI и BLoC зовут контракт, а не сам SDK.
- Не дёргать SDK напрямую из UI и не хранить его ссылку в BLoC.
- Контракты — без зависимости от Flutter и SDK.
