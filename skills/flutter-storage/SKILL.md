# Локальное хранение данных

Как сохранять и читать данные приложения на устройстве. Берёшь данные → смотришь, какого они типа → кладёшь в подходящее хранилище → так же читаешь. Объекты хранилищ берёшь из DI-контейнера.

Каждый тип живёт в своей папке под `common/storage/`: загляни туда — есть, бери и пользуйся; пусто — заведи минимальную структуру по образцу из `references/`.

## Три типа — когда какой

- **Простой KV** → `common/storage/preferences/`. Простые **несекретные** значения по ключу: тема, локаль, флаги, «онбординг пройден». Как завести — `references/preferences.md`.
- **Защищённый** → `common/storage/secure/`. То же «значение по ключу», но **секретное**: токены, ключи, пароли. Как завести — `references/secure.md`.
- **Локальная БД** → `common/storage/database/`. Много структурированных записей, коллекция растёт, нужен **запрос подмножества** (фильтр, сортировка, пагинация). Как завести — `references/database.md`.

## Границы

- **KV против БД** — по объёму и доступу. Ограниченный набор отдельных значений, читаешь целиком по ключу → KV. Много записей, и нужно доставать подмножество запросом → БД. Запах «пора в БД»: складываешь в KV JSON-список и каждый раз парсишь его, чтобы найти или обновить элемент.
- **Простой против защищённого** — секрет или нет. Токены, ключи, пароли → защищённый; всё прочее → простой.
