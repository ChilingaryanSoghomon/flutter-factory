# Серверный слой

Скилл про место и форму серверного слоя, а не про выбор конкретной библиотеки.

## Главное правило

Серверный API приложения всегда живёт в `lib/src/common/server/`, даже если конкретный запрос сейчас нужен одной фиче.

Фичи не берут server client напрямую. Фича описывает контракт в своём домене, BLoC работает с этим контрактом, а реализация репозитория обращается к `common/server/` и маппит результат в модели фичи.

## Структура

```text
lib/src/common/server/
├── api/           # группы запросов и server clients
├── dto/           # request/response модели сервера
├── mappers/       # перевод DTO в модели приложения или фичи
└── interceptors/  # optional: headers/auth/logging для выбранного клиента
```

Папки добавляй по мере необходимости. Не создавай пустые подпапки ради схемы.

## Стиль

- Не делай один общий `ApiService` на всё приложение.
- Группируй запросы по области сервера или ресурсу, чтобы файл оставался маленьким и понятным.
- DTO не протаскивай в BLoC, UI или domain фичи.
- Сырые исключения, status codes и ошибки клиента маппь на границе server/data слоя в доменный тип ошибки (`AppFailure` или текущий проектный аналог).
- Endpoints, окружение и общие лимиты бери из `AppConfig`.
- Реализации подключай через `common/core/di`, а не создавай клиент вручную в фиче.
