# Серверный слой

Скилл про границу приложения с внешними серверными источниками, а не про выбор конкретной библиотеки.

## Главное правило

Серверный слой всегда живёт в `lib/src/common/server/`.

Единица разделения — отдельный внешний источник: backend, provider или base URL. Даже если от источника нужен один запрос, у него своя папка.

Внутри server boundary внешние ответы, DTO, ошибки и особенности транспорта переводятся во внутренние типы приложения.

## Структура

```text
lib/src/common/server/
└── <source_name>/       # отдельный backend/provider/base URL
    ├── api/             # запросы и server clients этого источника
    ├── dto/             # request/response модели этого источника
    ├── mappers/         # перевод DTO во внутренние типы приложения
    ├── errors/          # перевод ошибок/status/envelope источника в ошибку приложения
    └── interceptors/    # optional: headers/auth/logging для выбранного клиента
```

Папки добавляй по мере необходимости. Не создавай пустые подпапки ради схемы.

## Стиль

- Не смешивай несколько внешних источников в одной папке.
- DTO и сырые response/envelope/error/status codes не выпускай за пределы server boundary.
- Ошибки каждого источника всегда маппь во внутренний тип ошибки приложения (`AppFailure` или текущий проектный аналог).
- Endpoints, окружение и общие лимиты бери из `AppConfig`.
- Реализации подключай через `common/core/di`, не создавай server client в местах потребления.
