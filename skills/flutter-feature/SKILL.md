# Фича: что это и как устроена

Что мы называем фичей, из чего она состоит и где её границы. Детали внутри слоёв — в детальном проходе.

## Что такое фича

Фича — **единица логического разделения (logical separation)**: связный кусок приложения, который мы решаем выделить отдельно. Определяет её именно отделённость, **а не состав** — иметь все слои фича не обязана. Это может быть осмысленная пользовательская область (авторизация, лента, профиль), один экран без логики, один `BLoC`, что держит state и не имеет интерфейса, — и всё это фичи.

Самое трудное — провести границу «где одна фича». Ориентир: за ней одна область, логически отделимая от прочего; обычно есть своё состояние (state) и правила, свой доступ к данным, её можно собрать и протестировать отдельно — но это **признаки, не требования**. Назрели внутри две такие области — это две фичи. Пока граница не проведена — скользко; как провёл — дальше фича делится по слоям почти механически. **Граница и есть главное решение.**

## Из чего состоит

Четыре слоя. Главный — **домен**: он описывает, *что* фича собой представляет; остальные три — его реализация.

- **Домен (domain)** — Flutter-free определения, «что есть фича»: контракты данных и сущности в `domain/`; события (events) и состояния (states) в `bloc/`.
- **Стейт-менеджмент (state management)** — логика `BLoC`: сам класс с обработчиками (в `bloc/`, рядом с событиями и состояниями) превращает событие и текущее состояние в новое. Здесь живёт логика фичи.
- **Данные (data)** — реализации контрактов фичи: обращаются к общим источникам данных, маппят результат в язык фичи и прячут детали источника.
- **UI** — экраны (`screens/`) и виджеты (`widgets/`): принимают события пользователя и по состоянию строят интерфейс.

Механизм состояния — `BLoC`; когда событий нет (например, внутри слушаем внешний источник и по нему меняем state) — берём `Cubit`.

Домен можно описать первым и закрыть тестами, а реализацию — следом. **Любой слой может отсутствовать** — фича берёт лишь нужное: один экран; один `BLoC` со state без интерфейса; в теории только данные. Подпапки внутри — только когда файлов реально много.

## Домен: чем оперирует фича

Flutter-free определения — то, что можно описать и закрыть тестами ещё до реализации:

- **состояние (state)** — что у фичи сейчас; всё, что видит UI, живёт здесь.
- **события (events)** — намерения пользователя или системы.
- **контракты (contracts)** — как фича ходит во внешний мир.
- **сущности (entities)** — доменные объекты фичи, когда простых переменных мало.

Каждое — по необходимости; чего не нужно, того нет.

## Поток (flow)

В одну сторону: **UI → BLoC → (опц. UseCase) → контракт репозитория (repository contract) → реализация (impl) → общий источник данных**. Обратно — данными и состоянием. UI шлёт событие (event), BLoC выдаёт состояние (state), наружу — только через контракт.

## Данные и внешние источники

Для фичи внешним считается всё, что лежит за её границей: серверный API, storage, platform-сервисы, SDK или общий объект из `common/`.

BLoC не берёт API-клиент, storage, SDK или platform-сервис напрямую. BLoC зависит от контракта репозитория из домена фичи. Репозиторий решает, откуда взять данные: из `common/server/`, `common/storage/`, `common/platform/` или другого общего источника.

`data/` фичи — не место для сырого server API слоя. Это место для реализации контракта фичи: вызвать нужный общий источник, смэппить результат в модель фичи и привести ошибку к доменному виду.

## Границы и общение

Главный инвариант: **фича меняет только своё состояние и только внутри своего BLoC, в обработчике события (event handler).** В чужое состояние она не лезет.

Отсюда — как фичи взаимодействуют (они **почти независимы**, но не наглухо):

- Чтобы повлиять на другую часть, фича **шлёт событие (event) в чужой BLoC** — обычно это BLoC фичи, что лежит **выше по дереву**. Импортировать чужой BLoC ради отправки события — нормально; меняет ли своё состояние, тот BLoC решает сам.
- Объект, через который общаются (особенно соседи одного уровня), сидит **выше обоих**. Это может быть BLoC (если нужен state-менеджмент), что-то проще вроде `ChangeNotifier`, или вообще обычный объект-хранилище, который просто держит данные для нескольких фич и состояние не меняет.

Что общее с другой фичей — поднимаем выше (в общий слой или в такой объект над фичами) и связываем при сборке.
