# Документация проекта: структура и жанры

Как разложить документацию так, чтобы человек **или агент** за минуту понял
проект: что это, как устроено по слоям и как работает конкретная фича. Держим
документацию в репозитории рядом с кодом (docs-as-code), чтобы она не устаревала.

## Роль скилла: и справочник, и генератор

Скилл одновременно:

- **справочник** — задаёт, какой должна быть структура `docs/` и формат файлов;
- **генератор** — при вызове разворачивает или обновляет документацию проекта.

При вызове он:

1. создаёт дерево `docs/` (универсальные зоны — ниже);
2. пишет `docs/README.md` — хаб: карта всех зон + правило «куда что класть»;
3. добавляет в `AGENTS.md` секцию `## Documentation` — короткие указатели на
   зоны и дисциплину (готовый блок — в разделе «Секция `## Documentation`»);
4. наполняет зону `app/` — карту архитектуры и user-flow не-очевидных фич (это исходное
   содержание скилла, см. раздел «Зона `app/`»).

**Повторный вызов — не с нуля.** Если `docs/` уже есть, дописываем недостающее и
приводим к формату, а не затираем существующее; секцию в `AGENTS.md` обновляем на
месте, а не дублируем.

## Универсальные зоны `docs/`

Есть в любом Flutter-проекте — формат этих зон **определяет данный скилл**.

```
docs/
  README.md            ← хаб: карта зон + «куда что класть»
  app/                 ← описание приложения
    architecture/
      overview.md      ← как устроено (зоны, слои, DI, навигация, запуск)
    features/
      README.md        ← индекс фич
      <feature>.md     ← поток не-очевидной фичи (не на каждую — см. ниже)
  current/             ← входная зона сырых идей
    current_context.md ← что в работе прямо сейчас
    <type>-<name>.md   ← одна мысль = один файл (idea / bug / feature)
  superpowers/         ← артефакты процесса (создаёт Superpowers, не этот скилл)
    specs/  archive/   ← + plans/ по мере появления активных планов
  testing/             ← стратегия тестирования
    README.md  runbooks/
```

Доменные зоны сверх этих (например `growth/`) — ниже, в разделе «Доменные зоны».

### Зона `app/` — описание приложения

Одна категория, две подпапки — два разных жанра (структура против сценария),
держим врозь, чтобы одно не тонуло в другом.

- `architecture/overview.md` — «**как устроено**»: зоны, слои, направление
  зависимостей, DI-гейт, навигация, флейворы и запуск. Одна карта на проект,
  читать первой. Источник содержания — skill `flutter-architecture` (зоны
  `common`/`features`, DI, навигация, флейворы). В конце — **карта связей фич**:
  многие фичи не имеют своего экрана и живут виджетом/панелью внутри фичи-хоста
  либо открываются по маршруту; таблица «кто кого встраивает и через что» +
  перечень фич без своего экрана даёт каждой фиче точку входа, не заставляя
  искать её грепом.
- `features/<feature>.md` — «**как работает по шагам**» (user flow): поток с
  развилками + опорные точки в коде. **Не на каждую фичу — только на ту, чей поток
  НЕ очевиден из кода** (несколько кооперирующих Bloc, нетривиальные развилки,
  грабли, «какой из двух путей настоящий»). Тривиальную фичу (один Bloc,
  самоочевидна) — строкой в индексе, без файла. Флоу дешевле сгенерировать из кода
  по запросу, чем держать протухающий снимок.
- `features/README.md` — индекс ВСЕХ фич со ссылками; у фич без файла — строка
  «что делает + где лежит» как точка входа.

Формат файла фичи и шаблоны — раздел «Правила файла фичи» ниже.

### Зона `current/` — входная зона сырых идей

Сырые текущие планы и заметки на самом раннем уровне (машинный вход, не
вылизанный человеческий документ).

- `current_context.md` — что **в работе прямо сейчас** (черновик, план, факты).
- **Одна мысль = один файл.** Дискретную мысль называем `<type>-<name>.md`, где
  `type` — один из `idea` / `bug` / `feature`; заголовок — `# Идея/Баг/Фича:
  <название>`. Для рабочих заметок по одной теме допустим файл-бэклог
  `<тема>-backlog.md` (например `subtitle_providers_backlog.md`) — это тоже
  черновик, а не постоянный реестр всего проекта. Файлы заводятся по мере
  появления мыслей; созрели → превращаются в задачу/план в `superpowers/`, файл
  из `current/` убираем.

### Зона `superpowers/` — артефакты процесса

`specs/` (design/spec) · `archive/` (завершённое); `plans/` появляется, когда
есть активный план. Папки и содержание создаёт процесс Superpowers, не этот
скилл — здесь только фиксируем, что зона существует, и её структуру.

### Зона `testing/` — стратегия тестирования

Стратегия по слоям + `runbooks/`. Только структура; наполнение — вне этого скилла.

## Доменные зоны (опционально) — не определяются этим скиллом

Проект может добавить зоны сверх универсальных — например `growth/` (ASO /
store-ops). Формат такой зоны задаёт **её собственный скилл**, а не `flutter-docs`
(для роста — `.agents/app-store-ops/skills/aso/`). `flutter-docs` лишь допускает
существование таких зон; `docs/README.md` ссылается на них как на точку входа
(для `growth/` — `growth/aso-state.md`). Их формат здесь не описываем.

## Секция `## Documentation` для `AGENTS.md`

`AGENTS.md` держит только указатели на зоны `docs/` — конкретика живёт в самих
зонах. Готовый блок ниже bootstrap вставляет в `AGENTS.md` как есть (шаг 3 при
вызове), чтобы не собирать указатели руками; ссылки в нём смотрят на зоны,
описанные выше (включая доменные).

```markdown
## Documentation

Working documentation lives in `docs/`; the entry point is `docs/README.md` (it maps
every zone and says where to put things). Main zones:

- `docs/app/` — what the project is and how it's built: the `architecture/overview.md`
  map (read it first) + `features/` (how each feature works, step by step).
- `docs/current/` — current raw plans and notes (mixed file types: idea / bug /
  feature, plus the active context). Drafts only: not an agent instruction file, not a
  project map, not a link hub.
- `docs/growth/` — ASO and store work (entry point `aso-state.md`).

`docs/superpowers/` artifacts (`specs/`, `plans/`, `archive/`) are working process
documents. Don't turn them into a mandatory reading path, and don't link to them from
code or product docs unless the current behavior can't be understood without the link.
```

## Дисциплина

- **Docs-as-code** — документы лежат рядом с кодом, чтобы не устаревать.
- **`current/` — только черновики**: входная зона сырых идей. Не инструкция
  агенту, не карта всего приложения, не история решений, не хаб ссылок.
- **`superpowers/` — не обязательный путь чтения**: задачу сформулировали,
  выполнили, проверили, закрыли (→ `archive/`). Не ссылаться на артефакты из
  кода, продуктовой документации или `current/`, если без ссылки не понять
  актуальное поведение.
- **`AGENTS.md` — только указатели**; конкретика живёт в `docs/`.
- Стабильная справка о проекте (архитектура, фичи) живёт в `app/`, а не в `current/`.
- **Документируй то, чего в коде дёшево не взять** — карту связей, точки входа,
  грабли, «почему так». НЕ пересказывай очевидное из `*_event.dart`/`*_state.dart`:
  раз агент может сгенерировать доку из кода — он может и просто прочитать код. И
  чем детальнее пофичевый флоу, тем быстрее он протухает; карта живёт дольше всех.

## Правила файла фичи (user flow)

- **Файл — только на не-очевидный поток** (см. «Дисциплину»): один файл = один
  поток, одна цель. Очевидную фичу не документируем — строка в индексе.
- **Один вход:** начинай с точки, где пользователь оказывается на экране фичи.
- **Развилки рисуй явно** (нет данных → кнопка → выбор пути А / путь Б).
- **Заканчивай ссылками на реальные файлы кода** — чтобы от текста прыгнуть в код.
- Коротко: сверху 2–3 строки «что это», в центре схема потока, снизу — код.

## Шаблон `docs/app/features/<feature>.md`

Внешний забор — четыре бэктика, чтобы вложенная схема потока (обычные три
бэктика) осталась внутри как есть. **Этот забор — обёртка для показа шаблона, а не
часть файла: реальный `<feature>.md` начинается со строки `# Фича:` и четырёх
бэктиков в себе не содержит.**

````markdown
# Фича: <Название>

<2–3 строки: что за фича и зачем.>

- **Экран:** `<Screen>` (`lib/src/features/<feature>/screens/...`)
- **Логика:** `<Bloc>` (`.../bloc/<feature>_bloc.dart`)
- **Хранение / источник:** `<Repository>` (`.../data/...`)

## Состояния экрана

| Статус | Что видит пользователь |
|--------|------------------------|
| ...    | ...                    |

## Главный флоу: <действие>

```
[Экран] → развилка? ──да──▶ ветка А
             │нет
             ▼
          ветка Б → событие Bloc → результат
             ├── успех → ...
             └── ошибка → ...
```

### Пошагово словами
1. ...
2. Развилка: путь А (`event(...)`) / путь Б (`event(...)`).
3. Успех → ... ; Ошибка → ...

## Другие действия (кратко)

| Действие | Событие |
|----------|---------|
| ...      | `...`   |

## Опорные точки в коде

- Экран: `...`
- События / состояния: `bloc/<feature>_event.dart`, `bloc/<feature>_state.dart`
- Данные: `data/...`
````

Опирайся на реальные события/состояния фичи (`bloc/*_event.dart`,
`bloc/*_state.dart`) — не выдумывай флоу, читай код.
