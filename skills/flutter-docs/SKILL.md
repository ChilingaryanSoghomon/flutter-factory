# Документация проекта: структура и жанры

Как разложить документацию так, чтобы человек **или агент** за минуту понял
проект: что это, как устроено по слоям и как работает конкретная фича. Держим
документацию в репозитории рядом с кодом (docs-as-code), чтобы она не устаревала.

## Два разных жанра — не смешивать

- **Архитектура** отвечает на «**как устроено**»: зоны, слои, направление
  зависимостей, DI, навигация, запуск. Одна карта на проект.
- **Фичи** отвечают на «**как работает по шагам**» (user flow): что за фича,
  поток с развилками, опорные точки в коде. По одному файлу на фичу.

Это разные жанры (структура против сценария) — держим в разных папках, чтобы
одно не тонуло в другом.

## Структура папки `docs/`

```
docs/
  README.md              ← хаб: ссылки на все разделы
  architecture/
    overview.md          ← карта проекта: зоны, слои, зависимости, DI, запуск
  features/
    <feature>.md         ← как работает фича по шагам (user flow)
```

- `docs/README.md` — оглавление-хаб: короткие ссылки на архитектуру и фичи.
- `docs/architecture/overview.md` — единая карта (см. skill `flutter-architecture`
  как источник содержания: зоны `common`/`features`, направление зависимостей,
  DI-гейт, навигация, флейворы и запуск).
- `docs/features/<feature>.md` — один файл на фичу.

`AGENTS.md` в блоке про документацию ссылается на `docs/architecture/overview.md`
и на `docs/features/`, чтобы агент нашёл вход. Не дублируй прикладную конкретику
в `AGENTS.md` — там только ссылки; сама конкретика живёт в `docs/`.

## Правила файла фичи (user flow)

- **Одна фича = один файл.** Один поток = одна цель.
- **Один вход:** начинай с точки, где пользователь оказывается на экране фичи.
- **Развилки рисуй явно** (нет данных → кнопка → выбор пути А / путь Б).
- **Заканчивай ссылками на реальные файлы кода** — чтобы от текста прыгнуть в код.
- Коротко: сверху 2–3 строки «что это», в центре схема потока, снизу — код.

## Шаблон `docs/features/<feature>.md`

```markdown
# Фича: <Название>

<2–3 строки: что за фича и зачем.>

- **Экран:** `<Screen>` (`lib/src/features/<feature>/screens/...`)
- **Логика:** `<Bloc>` (`.../bloc/<feature>_bloc.dart`)
- **Хранение / источник:** `<Repository>` (`.../data/...`)

## Состояния экрана

| Статус | Что видит пользователь |
|--------|------------------------|
| ...    | ...                    |

## Главный флоу: <действие>

​```
[Экран] → развилка? ──да──▶ ветка А
             │нет
             ▼
          ветка Б → событие Bloc → результат
             ├── успех → ...
             └── ошибка → ...
​```

### Пошагово словами
1. ...
2. Развилка: путь А (`event(...)`) / путь Б (`event(...)`).
3. Успех → ... ; Ошибка → ...

## Другие действия (кратко)

| Действие | Событие |
|----------|---------|
| ...      | `...`   |

## Опорные точки в коде

- Экран: `...`
- События / состояния: `bloc/<feature>_event.dart`, `bloc/<feature>_state.dart`
- Данные: `data/...`
```

Опирайся на реальные события/состояния фичи (`bloc/*_event.dart`,
`bloc/*_state.dart`) — не выдумывай флоу, читай код.
