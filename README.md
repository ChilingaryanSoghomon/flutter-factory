# flutter-factory

Агентский комплект для разработки Flutter-приложений.

## Вход

Агент автоматически видит только корневой `AGENTS.md`.
Если `AGENTS.md` указывает сюда, этот README — главный контракт Flutter-работы.

## Область

Комплект применяется к Flutter-работе:

- `new-app` — новое приложение;
- `feature` — новая фича или экран;
- `improvement` — улучшение существующего поведения;
- `bug` — исправление ошибки.

Backend, web и другие фреймворки позже должны быть отдельными комплектами.

## Рабочий порядок

1. Классифицируй запрос: `new-app | feature | improvement | bug`.
2. Составь `development-plan` по `templates/development-plan.md`.
3. Если данных не хватает, остановись и спроси пользователя. Не угадывай.
4. После утверждения плана используй `roles/tech-lead.md`.
5. Для реализации одной технической задачи используй `roles/flutter-engineer.md`.
6. После реализации проверь, выполнены ли критерии из плана и задачи.

Между стадиями есть точка контроля пользователя. Если результат неверный, сначала
исправляй артефакт нужной стадии, а не патчь код вслепую.

## Роли

- `roles/tech-lead.md` — переводит утверждённый план в технические задачи.
- `roles/flutter-engineer.md` — выполняет одну техническую задачу в пределах scope.

Перед работой в роли прочитай соответствующий файл роли.

## Скиллы

Скилл — это папка `skills/<name>/SKILL.md`. Источник правды — frontmatter:

```yaml
---
name: <kebab-case-имя>
description: <одна строка: когда применять этот скилл>
---
```

`description` объясняет, когда применять скилл. Ручной JSON-индекс не ведём.

- `skills/feature-structure/` — структура новой фичи/модуля.

Если скилл помечен как `Status: draft`, не считай его полным правилом.

## Шаблоны

- `templates/development-plan.md` — единый входной артефакт.
