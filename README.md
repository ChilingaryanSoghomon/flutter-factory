# flutter-factory

Комплект для работы с Flutter-приложениями.

Используй его, когда нужно создать, изменить, улучшить или исправить
Flutter-приложение.

У скиллов в начале файла есть `description`. По нему агент решает, когда
использовать файл. Ниже — каталог: короткий триггер на каждый скилл для
быстрого подбора; полный `description` — в самом файле.

## Правила

Единые правила кода — в `conventions.md` (нейминг, конструкторы, слои). Скиллы
их демонстрируют в примерах; единственный источник правды — этот файл.

## Процесс: рамки и метод

Замысел — пара. **flutter-factory** задаёт *рамки*: что строим и по каким правилам — архитектура, слои, навигация, хранение, тема, тесты (Flutter-знание). **superpowers** ведёт *процесс*: как это делается — дизайн, планирование, диагностика, реализация, ревью. Это вся его методология, а не отдельные шаги.

**Обязательное правило процесса.** Для каждой фазы ниже ты ОБЯЗАН выполнить
соответствующий superpowers-скилл как процесс, а не как справку. Это не указатель
и не рекомендация — это шаг, который нельзя пропускать. В частности: **запрещено
писать план или дизайн «от себя», по памяти или имитируя вид прошлых
документов.** Если ты собираешься создать план без запуска `writing-plans` —
остановись, это ошибка процесса.

- **Дизайн и проектирование** → ОБЯЗАТЕЛЬНО `superpowers:brainstorming`; облик (до кода) — раздел «Дизайн» ниже.
- **Планирование** → ОБЯЗАТЕЛЬНО `superpowers:writing-plans`. План создаётся ТОЛЬКО этим скиллом.
- **Реализация плана** → ОБЯЗАТЕЛЬНО `superpowers:executing-plans` или `superpowers:subagent-driven-development`.
- **Отладка** → ОБЯЗАТЕЛЬНО `superpowers:systematic-debugging`.
- Flutter-часть на всех фазах — по `conventions.md`; перед «готово» — `flutter analyze` и `flutter test`.
- Ручной/рантайм-smoke — на запущенном Android-эмуляторе (юзер поднимает через Android Studio). Сначала проверь `flutter devices`, не предполагай, что устройства нет.

**Если скилл не запускается одним вызовом `Skill`** (например, superpowers стоят
для Codex, а не как Claude-skill) — это НЕ повод пропустить процесс. Прочитай его
`SKILL.md` напрямую с диска (см. `INSTALL.md` про пути установки) и выполни шаги
буквально. Недоступность кнопки — не разрешение импровизировать.

Доменные скиллы (раздел «Скиллы» ниже) можно читать как справку и без superpowers.
Но фазы процесса — дизайн, планирование, реализация, отладка — без superpowers
выполнять нельзя.

## Скиллы

Знание по теме. Лежат в `skills/<name>/SKILL.md`; агент подбирает по теме задачи.

- **flutter-architecture** — границы и зоны приложения в целом, направление зависимостей, изоляция фич.
- **flutter-common** — общий слой: что переиспользуют несколько фич; верхнеуровневая карта общей зоны.
- **flutter-feature** — внутреннее устройство одной фичи: слои, поток данных, границы.
- **flutter-app-scaffold** — создание нового приложения или пересборка ядра; одноразовый каркас запуска.
- **flutter-navigation** — переходы между экранами через контракт навигации, охранники переходов, нав-элементы.
- **flutter-provider-scope** — область видимости провайдеров в дереве: где живёт зависимость (app-global vs сессия/маршрут), явные зависимости для переиспользуемых виджетов, re-provide через dialog/route/Hero.
- **flutter-server** — общение с внешними серверами: запросы, DTO, маппинг во внутренние типы, ошибки источника.
- **flutter-storage** — локальное хранение: настройки и флаги, секреты, структурированные данные в локальной БД.
- **flutter-theme** — оформление: тема, светлая и тёмная, токены, типографика, применение в UI.
- **flutter-observability** — видимость работы приложения: аналитика, логирование, отчёты о сбоях.

## Дизайн: облик до кода

Каким быть приложению — облик, цвет, типографика, иерархия, платформенные нормы.
Отдельный слой: не Flutter и не процесс, а знание о том, как должно выглядеть.
Решается до кода; на выходе — описание облика, которое затем превращают в план и
реализацию (`flutter-theme` и фичи).

- **frontend-design** — качественный фронтенд-дизайн: цветовые системы,
  типографика, уход от генерик-вида.
- **platform-design-skills** — платформенные конвенции (iOS, Android и далее):
  нормы облика и доступности под каждую платформу.

Ставятся отдельно для Codex/universal и Claude Code — см. `INSTALL.md`.

## Официальные скиллы

- **Тесты** → `dart-add-unit-test`, `flutter-add-widget-test`, `flutter-add-integration-test`, `dart-generate-test-mocks`, `dart-collect-coverage`.
- **Адаптивный layout** → `flutter-build-responsive-layout`, `flutter-fix-layout-issues`.
- **Локализация** → `flutter-setup-localization`.
- **Статический анализ** → `dart-run-static-analysis`.
- **Конфликты зависимостей** → `dart-resolve-package-conflicts`.
- **Паттерн-матчинг** → `dart-use-pattern-matching`.
